Why this page exists

Generic agent guidance does not survive production pressure.

The reference site explains why governance files matter. This project takes the same public-documentation idea and pushes it into implementation reality: scope packets, review packets, accessibility review, security review, release evidence, and verification scripts that fail when the repository drifts.

Bound the work before code starts

Scoping is a first-class artifact here, not a vague preamble. The repo forces teams to state risks, impacted files, evidence gaps, and out-of-scope areas before implementation.

Separate implementation from review authority

Review is intentionally stricter than implementation. Accessibility, security, and release truthfulness each get their own specialist lane when relevant.

Say what the repository can actually prove

The docs and scripts are explicit about starter-level limits. The workflow never pretends that prompts alone prove a production dashboard, auth stack, or deployment.

System surface

The repo is structured as an operating system for agent work.

Every layer has a clear job. The public site links to the real files instead of paraphrasing them into stale marketing copy.

Execution order

The workflow is intentionally strict.

This is not a “best practices” suggestion list. The repository defines a required order and the supporting files for each stage.

  1. 1

    Scoping

    Bound intent, risks, file impact, evidence gaps, and manual checks.

    Open scoping packet template
  2. 2

    Implementation

    Make the smallest correct change without broadening scope or weakening contracts.

    Open implementation prompt
  3. 3

    Review

    Call out scope drift, correctness risk, test gaps, and documentation drift.

    Open review packet template
  4. 4

    Specialist review

    Accessibility, security, and release honesty are separate review surfaces when relevant.

    Open specialist prompts
  5. 5

    Verification

    Run the required commands from requirements.toml and report real results.

    Open release verification script
  6. 6

    Documentation and release notes

    Describe only verified behaviour and disclose starter-level limits plainly.

    Open release prompt

Operational roles

Prompts are specialized because failure modes are specialized.

The repository does not collapse everything into one mega-prompt. Each role gets a separate prompt with a narrower contract and a clearer bar.

Scoping agent

Force explicit scope, risks, affected files, and verification expectations.

Implementation agent

Constrain changes to the approved scope and keep behaviour and docs aligned.

Review agent

Inspect the real diff and reject correctness drift, test gaps, and sloppy claims.

Accessibility review agent

Separate deterministic failures from manual judgement and avoid fake compliance language.

Security review agent

Inspect permissions, data paths, secrets, exports, and operational control boundaries.

Release agent

Block release language that says more than the repository can verify.

Proof rails

The repository is explicit about what is verified and what is not.

That honesty is a feature. Teams lose trust when workflow tooling quietly implies more maturity than the codebase actually proves.

Verified in this starter

  • Prompt structure and required file linkage
  • Workflow and packet presence checks
  • Unsupported-claim scanning in key docs
  • Starter manifest generation
  • Worked example and teaching asset presence

Not verified by this starter

  • Live dashboard UI behaviour
  • Production auth or authorization flows
  • Export correctness for CSV, PDF, or screenshots
  • Operational durability of background jobs
  • Deployment readiness of an application stack
